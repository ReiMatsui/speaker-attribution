# 引き継ぎプロンプト（新しいチャットの冒頭に貼る）

あなたは、対面会議をリアルタイムに「誰が・何を」話したか日本語で文字起こしする
研究/実装プロジェクトを引き継ぎます。作業フォルダは `~/speaker-attribution`
（Mac, Apple Silicon M4 Pro, uv管理, Python 3.13）。まず `docs/` 配下
（RESEARCH_LOG.md / SUMMARY.md / LANDSCAPE.md / この HANDOFF.md）と CLAUDE.md を読んでください。

## ゴール（不変）
対面会議を、低遅延・リアルタイムで「誰が何を言ったか」に文字起こしする（日本語）。

## ここまでの結論（重要）
- 当初は「1人1マイク＋自前の漏れ込み除去ゲート（VAD→チャンネル間相関 or 声紋）」を研究。
  合成データとAMI実会議で検証し、「強いASRほど漏れ込みで崩れる→ゲートが効く」ことを確認。
  ただし**良いヘッドセット＋強ASRなら生でもそこそこ動き、ゲートは"仕上げ掃除"**という実データ知見も得た。
- リアルタイム化を実機検証した結果、**ローカルWhisper（mlx-whisper/parakeet）は日本語ライブで品質が不安定**
  （幻聴「ご視聴ありがとうございました」、繰り返し暴走、ラグのムラ）。parakeetは日本語非対応。
- **最終的に Soniox のリアルタイムAPI（日本語・話者分離内蔵）を本線に採用**。
  1本のマイクで「話者1/話者2…」の who-said-what が、低遅延・高品質で取れることを実機確認済み。
  → 多マイク・自前ゲート・デバイス同期は不要になった。研究のゲートは「同時発話が激しい/各人マイク固定で
  堅くしたい」場合の**オプションの上乗せ**という位置づけ。

## 現在の本線ツール
- `offshelf/live_soniox.py` … 1マイク＋Soniox（日本語・話者分離）リアルタイム議事録ツール。**これが本命。**
  - 必要: `uv add websockets sounddevice` / `export SONIOX_API_KEY=...`（Sonioxは残高必要・少額チャージ）
  - 実行: `uv run python offshelf/live_soniox.py`（`--wav` でファイル擬似ライブも可、`--out` で保存先指定）
  - 機能（2026-06-10実装、モックサーバで検証済み・実機ライブ確認は未）:
    ① Markdown議事録を `transcripts/日時.md` に自動保存（発話確定ごとに原子的書き出し＝クラッシュ安全）
    ② 実行中に「1=松井」と打つと話者1を実名化（過去の発言も遡って置換）
    ③ 話者ごと色分けライブ表示、確定前テキストは薄く表示、`[MM:SS]` タイムスタンプ付き
    ④ HTML版も同時保存。起動時にブラウザ自動オープン（ライブ中2秒ごと自動更新、--no-openで無効）
    ⑤-b 自動登録・最終形（2026-06-10夜、簡素化版）。判定は2経路のみ:
       ① 即時判定: 単発声紋が強一致(--vp-match=0.75 ＋2位差0.05)した時だけその場で人物確定
          （Sonioxラベルの入れ替わりもこれで補正される）
       ② それ以外（弱一致・未知・僅差の区別なし）は3発話バッファ: 一貫した3発話(全ペア>=0.62)を
          束ねたプロファイルで「既存人物に合流(>=0.70) or 新規人物N」を確定
       不変条件: 確定済みの人物キーは書き換えない。遡及置換は「話者N→人物N」の昇格のみ。
       実名(1=松井, 人物2=田中)のみvoices.jsonに永続化、匿名はセッション限り。--vp-no-autoで無効。
       経緯: 弱一致の即吸収(floor=0.65)が「3人目が既存人物に吸収」を起こし、二段確認・様子見等の
       継ぎ足しで複雑化＋「人物キーの遡及置換」バグも発生。即時判定を強一致のみに絞ることで
       特殊ケースを全廃し、性能は同等(検証: 素66.3%→簡素化版81.0%、Soniox完璧時の害1.9%、
       過酷条件72.8%)。声紋実測: 単発の同一人物類似は平均0.70とノイジー、3発話束ねで0.81。
    ⑤-c セッション自動校正（2026-06-10深夜、多人数・同室バイアス対策）:
       同じ部屋・同じマイクだと全員の声紋類似が一様に上振れし（部屋バイアスβ）、固定しきい値が
       破綻する（シミュレーション: β=1.0で現行37%と素ラベル65%を下回り全員が1〜2人に融合）。
       対策: ラベル内類似(=同一人物らしさ)とラベル間類似(=別人らしさ)の中央値を会議中に実測し、
       しきい値をその中間に自動配置。ただし max(既定値, 校正値) で「厳しくする方向にのみ」働く。
       検証: きれいな部屋で現行と同一(84%)、β=0.6で62.9→72.2%、β=1.0で36.6→66.8%。
       終了時統計に「部屋の声紋分布: 同一X/別人Y→しきい値…」が表示され部屋の状態を確認できる。
    ⑤-d 汚染・新人検出の検証（2026-06-11、検出漏れ50%＋遅参加新人モデル）:
       懸念「重なり検出は片側のみ→混合声紋がプロファイルを汚染」は実在するが、3発話一貫性
       ゲートが混合サンプルを濾過するため実害は小さいと判明（遅延採用①を試したが効果なし→不採用）。
       採用: 即時マッチ時にバッファをリセットしない（②）。リセットすると新話者の3発話が
       貯まらず吸収されやすい。遅参加者の独立ID獲得64→66%/74→78%、コードも1行減。
       注意: YouTube再生音でのテストは全話者が同一再生チェーンを通るため実会議より分離が困難
       （部屋バイアスの強化版）。生身の複数話者では数値が改善する見込み。
    ⑤-e 声紋モデル切替（2026-06-11、--vp-model ecapa 実験版）:
       Resemblyzer(2019世代)の純粋な聞き分けは分布が重なる(同一5%点0.53 vs 別人95%点0.59)。
       ECAPA-TDNN(speechbrain)は実音声でほぼ完全分離(0.18 vs 0.17)＋計算10倍速(27ms)。
       ただし実音声の混合(重なり)をECAPAは成分話者と強くマッチさせる癖があり(97%が即時超え、
       雑音下で主話者を抽出する訓練の副作用)、通しシミュレーションでは優劣つかず。
       実地A/B（ユーザーのYouTube多話者テスト）でecapaの体感が明確に良かったため
       **既定をecapaに変更**（2026-06-11）。依存未導入なら自動でresemblyzerにフォールバック。
       しきい値はモデル別既定(DEFAULTS)、voices.jsonに_modelを記録し混在を防止。
       注: モデルを切り替えると既存voices.jsonの声紋は読み込まれない（再登録が必要）。
    ⑤ 声紋プロファイル方式の話者特定（2026-06-10、オンラインクラスタリングから再設計）:
       「1=松井」で話者1の直近声紋(Resemblyzer)を voices.json に登録。以降と次回以降の会議は
       声の照合だけで自動的に実名表示。未登録の声はSonioxラベル(話者N)のまま。
       ロジックは「発話(1秒以上)→声紋→登録者の中で一番似ている人(類似度--vp-floor=0.60未満は未知)、
       短い相槌はラベルの直近判定に追従」だけ。台帳が固定なので誤判定が伝播しない。
       同時発話対策: 他の発話と時間が重なった発話は声が混ざるため声紋判定をスキップし
       直前対応を維持（誤マッチで対応表を汚さない）。1位と2位が僅差(--vp-margin=0.05未満)の
       判定も保留。重なった瞬間の文字の取り違え自体は1本マイクの原理的限界（残る場合は
       会議後の非同期API再処理 or 1人1マイクが次の手）。
       --no-vp で無効化。「fix 3=1」で表示の統合。終了時にレイテンシ統計。
       経緯: 当初はオンラインクラスタリング(台帳をその場で学習)で実装したが、
       話者分裂→自動統合→誤統合の固定化→脱出口…と補正ルールが増殖（誤りが台帳に蓄積・伝播
       するのが根本原因）。台帳固定のプロファイル方式に再設計して大幅に簡素化した。
       終了時にレイテンシ統計を表示。「fix 3=1」で誤分裂を手動統合も可。
       注: 現行Soniox API(v4)に声紋登録機能は無い（旧APIのみ）、RT分離は15人上限・非同期より精度低。

## 研究コード（src/spkattr/）— 必要時に使う
VAD / crosstalk(GCC-PHAT) / embedding(Resemblyzer) / pipeline / streaming(StreamingGate) / asr(Whisper) / synth / evaluate。
評価スクリプトは scripts/（eval_separation, eval_ami, eval_streaming_gate 等）。AMI実データDL済み。

## 重要な落とし穴（ハマりどころ）
- zshに `# コメント` 付きコマンドを貼ると引数エラー → コメントなしで1行ずつ実行。
- gitがマウントのロックで失敗することがある → `rm -f .git/*.lock` してからコミット。
- parakeetは日本語非対応（英語/欧州語）。kotoba-whisper-mlx は mlx_whisper と非互換で空出力。
  → 日本語ローカルは whisper-large-v3-turbo（不安定）、本命はSoniox。
- iOS Safariのマイクは**HTTPS必須**（meeting_serverは自己署名証明書を自動生成、各iPhoneで許可が必要）。
- macOSのターミナルに**マイク権限**が必要（無いと無音）。
- **APIキーはチャットに貼らない**（露出したら失効・再生成）。

## 方針
精度の数字より「実際に分離して議事録として使えるか」を優先。良ければ妥協して実用化を進める。
日本語で簡潔に、要点先出し。作業前に計画を共有し、長い調査や実装の前に確認を取る。
