# 研究ログ：1人1マイク対面会議の話者帰属つきリアルタイム文字起こし

最終更新: 2026-06-09
このドキュメントは、何を目指し、何を試し、何が効いて何が外れたか、そしてなぜそうなったかを、
誇張も省略もせず記録するためのものです。失敗も今後の知見として残します。

---

## 1. 究極のゴール（ぶれさせない）

対面の会議・議論を、**低遅延（できるだけリアルタイム）で「誰が何を言ったか」のテキストにする**こと。

話者帰属（誰の発話かを当てる）は、それ自体が目的ではなく、文字起こしを正しくするための
**関門（漏れ込み除去ゲート）**として位置づける。1人1マイクを前提とし、各マイクの音から
「持ち主の発話だけ」を残してASRに渡す。最終的な評価指標は、話者を考慮した単語誤り率
（cpWER）とレイテンシ。

この前提（1人1マイク・対面・低遅延）は固定し、揺らさない。

---

## 2. 現在のシステム構成

```
各マイク信号
  → VAD（発話区間検出, エネルギーベース）
  → チャンネル間相関（GCC-PHAT）で漏れ込み判定
  → 声紋照合（Resemblyzer, enrollmentあり/なし）  ※後述の通り効果薄で既定オフ寄り
  → 連続マスクして本人時間以外を無音化
  → Whisper（ローカル）で一括文字起こし
  → 話者ラベル付きトランスクリプト
```

---

## 3. 評価の考え方の変遷（ここが研究の信頼性の核）

評価指標と正解の定義を、信頼できる方向へ3段階で改めた。これ自体が重要な学び。

**段階1：窓単位の precision/recall（合成正解）。**
500ms窓ごとに「本人発話を当てたか」を測った。動作確認には十分だが、正解を自前合成の
エネルギー閾値で作っており、閾値次第で数字が動く。研究の主張には弱い。

**段階2：tinyモデルのWhisper出力を正解にする（誤り）。**
クリーン音声のWhisper出力を正解とし、漏れ込み版と比較した。同一モデルなのでWhisper自身の
誤りが相殺され「漏れ込み＋ゲートの影響」だけを測れる利点はある。しかし**絶対精度の評価としては
不適切**。tinyは性能が低く、それを正解扱いするのは信頼できない。ユーザー指摘により是正。

**段階3：LibriSpeechの公式トランスクリプト（人手参照）を正解にする。**
正解からWhisperを排除し、人手の参照テキストでWERを測る方式に変更。これが本筋。
別話者を dev-clean から自動抽出して使う（`scripts/eval_librispeech.py`）。

**確立した原則（CLAUDE.mdにも記録）：** 正解がきちんと定義された実データセットで評価する。
合成データは「漏れ込みゲートの効果を切り分ける内部検証」に限定して併用する。

## 4. 実験の時系列と結果（正直に）

### 4.1 統合プロトタイプ
VAD＋チャンネル間相関＋声紋照合を1本に統合。合成クロストーク（1人録音に遅延・減衰を足して
混ぜる）で動作確認。漏れ込み棄却率は高い一方、本人発話の取りこぼし（recall）が低かった。

### 4.2 VADのバグ（修正済み）
当初、500ms窓ごとにVADのノイズ床を推定していたため、窓全体が発話だと「床＝発話レベル」となり
何も検出できなかった。チャンネル全体で一度VADを計算するよう修正し、recallが大きく改善
（合成データで 0.21→0.57）。**教訓：窓単位の正規化は連続発話で壊れる。**

### 4.3 ASR統合
本人と判定した区間だけWhisperにかけ、話者ラベル付きトランスクリプトを出力。
CPU・tinyでRTF 0.26（実時間より速い）。ゴールの形（who-said-what）に到達。

### 4.4 評価をLibriSpeech公式参照に切替（段階3）
別話者4名・隣どうし2秒重なる会話・漏れ込み −12dB・Whisper base で、全体WER **30%**。
クリーン下限は **約4%**。両者の差（約26pt）が「漏れ込み＋本研究の処理」が持ち込む誤り。

### 4.5 分離評価（ablation）と決定的な診断
誤りの内訳を切り分けた結果、重要な事実が判明した。

**「漏れ込み」はほぼ犯人ではなかった。真犯人は「ゲートの音の切り刻み」だった。**

診断：漏れ込みなしのクリーン音声でも、**ゲート区間で断片に切ってASRすると WER 28.7%**
（クリーン全体なら 3.7%）。つまり 500ms 窓で音をぶつ切りにすると、Whisperが文脈を失い
単語を落とす。漏れ込み自体が足す誤りは差し引き約1ptに過ぎなかった。

### 4.6 効かなかったアイデア（失敗の考察）

最初の仮説は「ASRに漏れ込み入りの生音を渡しているのが悪い。音をクリーン化すれば改善する」
だった。これは**外れた**。

- **時間周波数の支配チャンネルマスク（tf_channel_mask）：WERが悪化（30%→33%）。**
  各T-F点を最大エネルギーのマイクに割り当てるソフトマスク。理屈は妥当だが、マスクが
  スペクトルに歪み（ミュージカルノイズ）を入れ、Whisperは弱い漏れ込みより**マスク由来の歪み
  に弱い**。きれいに消そうとして、かえってASRが嫌う人工物を作った。

- **最小二乗FIRキャンセル（lsq_cancel）：ほぼ無効（変化なし）。**
  他チャンネルを参照に漏れ込みを線形予測して減算する。だが参照チャンネル自身も本人の声を
  含み、各チャンネルは自分の話者が支配的なので、他チャンネルからの予測が弱く、減算量が
  ほぼゼロになった。クロストークの相互混入を無視した素朴な定式化の限界。

- **ソフトな無音化（floorを残す）：大崩壊（WER 154%）。**
  境界の歪みを嫌って非本人区間を完全に消さず減衰に留めたら、漏れ込みが通ってWhisperが
  他人の声まで書き起こし、挿入誤りが爆発した。**漏れ込みは中途半端に残すと致命的。**

- **全体ASR→単語タイムスタンプで取捨選択（attribute_words）：悪化（39.8%）。**
  混合チャンネル全体を文字起こしすると他人の漏れ込みまで全部テキスト化され、単語単位の
  時間フィルタでは取り切れなかった。

**考察：** 「音をクリーン化する」より「**音を断片に切らず、本人時間以外を無音化した連続音声を
一括ASRする**」方が圧倒的に良い。Whisperは多少の弱い漏れ込みには頑健だが、断片化と
スペクトル歪みには弱い、という性質が一貫して効いている。

### 4.7 効いたアイデア（採用）
- **連続マスク方式のASR：** 本人発話と判定した時間以外を無音化した「全長の連続音声」を
  チャンネルごとに作り、一括でASR。断片化による単語落ちを防ぐ。
- **recall優先のゲート：** 声紋照合の閾値を下げ（または外し）、VADを緩める。漏れ込みが
  ASRに足す誤りは小さいので、多少通してでも本人発話を落とさない方が得。
- **マージ幅・パッドの拡大：** 本人区間を広めにつなぎ、発話を分断しない。

結果、−12dB で **WER 30% → 13%**（誤りを半分以下に）。

### 4.8 漏れ込み量スイープ（効く領域の特性化）
漏れ込みの強さを振って、ゲートなし(B)とゲートあり(C)、クリーン下限(A)を比較。

| 漏れ込み | A クリーン下限 | B ゲートなし | C ゲートあり（本研究） |
|---|---|---|---|
| −12 dB（弱） | 4% | 6% | 13% |
| −6 dB（中） | 4% | **130%** | **14%** |
| 0 dB（等音量） | 4% | 278% | 233% |

- 弱い漏れ込みは漏れ込みが静かすぎてWhisperが拾わず、ASR単体でほぼ解ける。ゲートはむしろ
  本人発話を少し削って損をする。
- 中程度の漏れ込みでゲートなしは他人の声まで書き起こして崩壊し、本研究が大きく回復させる。
- 等音量(0dB)はゲートありでも崩壊する最難領域。

### 4.9 1本マイク vs 1人1マイク（路線の比較）
「最終的に1本マイクで済むならその方が良い」という要望を受け、1本マイク版（全員を1本に混ぜ→
Whisper→声紋で各単語を担当者へ振り分け）を作り、同じターンテイキング会話で比較した。
結果（base, 公式参照）：クリーン下限 4% / **1本マイク 21% / 1人1マイク 13%**。
1本でも崩壊せず実用圏だが、同時発話で声紋が混ざり1段落ちる。差はほぼ「重なりの瞬間」に集中。

**決定：主軸は1人1マイク**（帰属が堅くデータも作りやすく、まず成功させやすい）。
1本マイク路線は手軽さゆえ将来の有力な簡素化案として保持する（本ログとCLAUDE.mdに記録）。

## 5. 妥当性への正直な注意（threats to validity）

研究として弱い・要改善の点を隠さず列挙する。

1. **合成データである。** 実際の部屋の反響、マイクの周波数特性、非線形性、実際のクロストークの
   畳み込み的性質を簡略化している。漏れ込みは「遅延＋減衰」の線形モデル。実録音はもっと複雑。

2. **WERであってcpWERではない。** いまの評価は話者ごとのWERを集計している。**他人の発言を
   自分のものとして取り込む「誤帰属」を十分に測れていない**おそれがある。漏れ込みが静かで
   Whisperが拾わない領域では誤帰属が表面化せず、ゲートの価値が過小評価されている可能性。
   標準のcpWERへ移行すべき。

3. **漏れ込み量の現実性が未検証。** 0dB（等音量）は明らかに非現実的（後述）。逆に −12dB が
   現実かどうかも実測で確かめていない。実際の漏れ込み量はマイク種別（ヘッドセット/ピン/卓上）
   と距離で決まる。ここは実機 or AMIコーパスで grounding が必要。

4. **オフライン・英語・small以下のモデル。** まだリアルタイムではなく、言語も英語、ASRも軽量
   モデル。レイテンシは未計測。

5. **enrollmentの扱い。** 現状の良い結果は声紋照合を外した（または緩めた）設定で出ている。
   「別人を別人と認識」という当初の声紋的な目標と、実利（recall優先）の間で整理が要る。

## 6. ゴールに対する現在地

到達したこと：1人1マイクの合成会議で、公式参照テキスト基準のwho-said-what文字起こしを
出力でき、漏れ込みが効く領域では誤りを大幅に抑えられることを示した（−6dBで 130%→14%）。
評価を信頼できる実データ正解へ移行し、効く手法（連続マスク）と効かない手法（音のクリーン化）を
切り分けた。

未到達：リアルタイム化（ストリーミング・レイテンシ計測）、cpWERでの誤帰属評価、実データ
（AMI）での検証、相槌・割り込みの扱い。

## 7. 重要な再フレーミング：対面議論で本当に起きる条件は何か

スイープの 0dB（全員が同音量で常時重なる）は、**対面の議論では基本的に起きない**。
理由は2つ。

第一に、1人1マイク（特にヘッドセットやピンマイク）では、本人の声が自分のマイクで最も大きく
入り、他人の声は距離の分だけ小さく漏れる。全員が各マイクで等音量になるのは、近接マイクの
前提と矛盾する（それは卓上の遠いマイクの状況に近い）。

第二に、対面の議論は基本的にターンテイキングで進み、重なるのは相槌や短い割り込みなど
一時的・部分的なもの。全員が長時間まるごと重なることはまずない。

したがって、**0dB・全面重なりは「最難ストレステスト」であって、設計目標ではない**。
現実の動作点は「弱〜中程度の漏れ込み＋ターンテイキング＋短い相槌の重なり」。この領域に
評価を寄せるべきで、ここを取り違えないことが重要。

**正直な含意：** 現実が弱い漏れ込み寄りなら、ASR単体でもかなり解け、本研究の上積みは
小さくなる。本研究が明確に効くのは「漏れ込みが無視できない中程度の領域（卓上マイク・
感度の高い環境・近接した話者）」と、「相槌/割り込みを正しく当て分けて誤帰属を防ぐこと」。
後者は raw WER では見えにくいので、cpWER での評価が次の鍵になる。

## 8. 次の一手（優先順）

1. **評価をcpWERにし、相槌・割り込みの誤帰属を測れるようにする。** 現実の動作点
   （弱〜中漏れ込み＋部分重なり）でのゲートの真価はここで初めて見える。
2. **漏れ込み量の grounding。** AMIコーパス（実1人1マイク＋正解）で実際の漏れ込み量と
   性能を確認する。
3. **ストリーミング化とレイテンシ計測。** 対面会議の本丸。
4. **等音量フロンティア（最難）への対処** は優先度を下げる（非現実的なため）。必要なら
   target-speaker抽出を検討。
```
